# Roll'Em — Hiring Plan

Owner: CEO. Last updated: 2026-07-07 (ROL-1).

## Principle

Hire slow, against a concrete bottleneck — never ahead of one. Every hire must map to a milestone we cannot ship without them. Until then, the founding team stays lean.

## Current team

| Role | Agent | Status |
|---|---|---|
| CEO | CEO | Active. Strategy, priorities, board comms, hiring. |
| Founding Engineer & CTO | CTO | **Hired 2026-07-07.** Owns all technical execution: architecture, game engine, web UI, testing. |

## Hiring roadmap

Hires are gated on milestones, not dates.

### Phase 1 — Playable MVP (now)

No additional hires. The CTO builds the dice game MVP end to end. The CEO owns product direction and playtest feedback.

### Phase 2 — Polish & first users (trigger: MVP is playable in the browser)

1. **UX Designer** — first hire after the CTO. A dice game lives or dies on feel: readable game state, satisfying rolls, obvious choices. Hire when the MVP works but doesn't feel good.
2. **QA** (optional) — only if rule/scoring bugs keep escaping the CTO's test suite.

### Phase 3 — Growth (trigger: game is fun per playtests and we want players)

3. **CMO** — content, launch, community. Hiring earlier would mean marketing a game that doesn't exist.
4. **Second Engineer** — only if the roadmap forks (e.g., multiplayer + mobile in parallel). One workstream = one engineer.

## What we deliberately do NOT hire

- **CFO** — no revenue, no spend complexity yet.
- **PM** — CEO owns product until team size makes that a bottleneck.
- **DevOps** — a browser game MVP needs no dedicated infra role.

## Process for each hire

1. CEO writes a one-paragraph thesis: what bottleneck this hire removes and what milestone unblocks.
2. CEO creates the agent (role, instructions, reporting line) and a first assignment issue.
3. New agents report to the CTO if technical, to the CEO otherwise.
